## Arquitetura

```
[Painel Next.js (Vercel)]  ─HTTP─►  [Evolution API VPS:8080]  ─sticky proxy─►  WhatsApp
        │                                    │                                    
        ├── /api/chips/*                     ├── /instance/* (Baileys)            
        ├── /api/chips/proxy-heal (cron)     ├── /proxy/set                       
        └── /api/health                      └── redsocks (:9090)                 
                                                                                  
                          [Chatwoot (Hetzner :3000)]
                          ├── Inbox 84 + outros 30 inboxes (Channel::Api)
                          └── webhook → vip-players (auto-resolve)
```

- Painel = orquestrador Next.js. NÃO mantém estado.
- Evolution = roda Baileys, mantém sessões WhatsApp ativas.
- Proxy = entregue inline na URL via `/proxy/set/{name}` (rotina sticky-session com sufixo `_country-br_session-{name}-{ts}` para IPRoyal default OU credencial manual livre).

## Fluxo de connect ATUAL (após rewrite de hoje)

`lib/evolution.ts::createInstance` agora faz:

```
1. POST /instance/create  { qrcode: false }              ← cria sem iniciar Baileys
2. POST /proxy/set/{name} { ... }                        ← persiste proxy
3. GET  /proxy/find/{name}                               ← VERIFICA que ficou salvo
4. GET  /instance/connect/{name}                         ← Baileys inicia COM proxy → pairing code
5. Se não veio code: poll 1s × 6s
6. Refresh final via /instance/connect                   ← code "fresco" pro user (~40s validade WA)
```

Em qualquer falha tem 1 retry (delete + recreate). Erros são logados com flags `_firstError`, `_secondError`, `_proxy_set`, `_recovered_via_poll`, `_refreshed`, `_retried`.

Confirmado via `/proxy/find/{name}` — chips criados via inline proxy retornam `null`. Tive que reescrever pro fluxo create→setProxy→connect.

### 2. Existiam 5 instâncias sem proxy nenhum

`VIPA210`, `SA39`, `SA90`, `SA23`, `SA73` — chips antigos que nunca tiveram proxy persistido. Estão (ou estavam) leakando o IP do servidor Hetzner pro WhatsApp anti-fraude.

### 3. 4 chips conectados HOJE também ficaram sem proxy

`VIPA210`, `VIPA211`, `VIPA214`, `VIPA215` — criados com o "fix" inline que não funcionava. **Continuam leakando agora**. Precisam de setProxy manual ou reconexão pelo novo código.

### 4. Causas distintas das "Falhas ao gerar pairing code"

| Tipo | Sintoma | Solução |
|---|---|---|
| **Baileys lento** | `qrcode.count: 0`, 7-9s | Resolvido com retry+poll |
| **Code expirado** | "Pairing code inválido" no app WA | Resolvido com refresh final |
| **WhatsApp rate-limit MSISDN** | Falhas consecutivas no mesmo número | Esperar 15-20 min, fora do alcance |
| **Proxy lento/dead** | `set_proxy_failed: TimeoutError` ou `no_pairing_code_after_poll` | Pegar IP novo no checker da marketbet |
| **Session limit do proxy provider** | Falha em sequência após N chips | Mudar credencial ou aguardar |
